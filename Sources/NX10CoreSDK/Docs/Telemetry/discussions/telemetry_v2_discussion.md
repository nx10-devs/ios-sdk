Details
Consolidated Touch Event Specification: The team is moving away from separate touch KB, touch, and keyboard events, consolidating them into a single touch event moving forward. The new API includes versioning, introduced by Sam Marshall, to handle future changes without breaking compatibility. The touch ID, based on Unity's gesture tracking, is desired to identify related touch points along a movement path, which would simplify logic implementation for David Foster.
Touch Event Data Attributes: The touch event will include a type indicating whether the action is a down action (finger placed), a move, an up action (finger removed), a stationary hold, or a canceled event. The canceled event is similar to an up event, occurring when the system stops tracking the touch, which may or may not be relevant for iOS. Touch object details, currently only relevant for the keyboard, are used to categorize key presses, such as submit/go buttons, backspace, letters, numbers, symbols, and utility functions like searching for a GIF.
Touch Location and Radius: Touch location must be recorded in millimeters, normalized to the bottom-left corner of the screen as 0, 0. The touch radius, also standardized in millimeters, is needed to estimate the size of the finger and the pressure applied to the screen. David Foster expects that a touch event, including a movement, will generate a new touch ID for each distinct contact (e.g., touching the phone twice will result in two IDs), with each touch ID tracing the movement path.
Data Sampling and Filtering: Although the device samples data at a high frame rate (e.g., 60 Hz), the data should be filtered before being sent to the API to reduce the volume of "move" actions. The desired filtering rate is closer to 30 Hz, ensuring a smooth line without excessive noise or redundant points, and the SDK will need to define a filter for this. Determining when a touch is genuinely stationary requires setting a threshold, such as not moving more than three pixels in one second.
SDK Implementation and Keyboard Kit: Warrd Adlani noted that the SDK currently relies heavily on Keyboard Kit, which responds specifically to key actions (down, up, and key type). They will investigate creating an SDK system to watch the entire screen, confirming the feasibility of recording all motions on the screen simultaneously with keyboard motions. It was agreed that for keyboard touches, the team is primarily interested in down/up actions, not move actions, unless swipe functionality is introduced.
Coordinate Standardization Challenges: A primary challenge is standardizing coordinates, as the keyboard uses its own coordinate space (top-left 0, 0) which is opposite to the requested bottom-left 0, 0 of the screen. Converting pixels into millimeters is essential due to differing screen resolutions (dots per inch or DPI) across devices. Normalizing to millimeters is necessary so that movement speeds can be accurately calculated regardless of the screen size or resolution.
DPI and Millimeter Conversion Responsibility: The SDK is responsible for calculating and converting coordinate data to millimeters, relying on device-specific properties like DPI, which Sam Marshall suspects is accessible on iOS. Warrd Adlani will investigate obtaining the necessary DPI information from the device to perform the millimeter conversions. David Foster confirmed that the ideal output is the screen's XY coordinates in millimeters, with the origin (0, 0) being the bottom left, regardless of whether the input comes from the keyboard or elsewhere.
Handling Phone Orientation: The team needs to determine how to handle landscape and portrait modes, as the keyboard size and shape change with rotation, which affects the coordinate system. David Foster will research the stats they need to calculate to decide if the bottom-left 0, 0 should remain fixed regardless of rotation or if it needs to change. Sam Marshall confirmed the new API is deployed on stage for Warrd Adlani to begin testing.
Keyboard State and Session Tracking: New, lower-priority event types are being introduced, starting with the keyboard state (up or down), which can be used to define the start and end points of a session for calculation purposes. Warrd Adlani confirmed that the SDK is already sending analytics for the keyboard appearing and hiding. The other new event type is a keyboard-specific event for reporting the number of characters erased by a backspace touch.
Text Correction Reporting: It is crucial to report events related to text correction, such as autocorrect, to measure typing accuracy. The team will need to differentiate between assertive automatic autocorrect (e.g., when the spacebar is pressed) and suggested changes (e.g., when the user taps on a suggestion). Warrd Adlani confirmed they were working on fixing autocorrect reporting and will investigate the capability to undo autocorrect in iOS.
API Testing and Error Handling: The API's validation is designed to log errors internally but not return errors to the user's phone, which prevents upsetting the device. Sam Marshall will share the Axiom link so Warrd Adlani can view logs and errors. Warrd Adlani will attempt to run the API locally to observe the direct output logs and see what data passes through to David Foster's pipeline, which Sam Marshall confirmed is possible.
Ingestion API and Testing Tools: The ingestion API runs on Node and Express, and can be run locally by installing the NPM dependencies and running the `npm rundev` command in the ingestion package. The control plane, which requires Docker and a Mongo container, is not necessary for testing the ingest API. Sam Marshall also uses Bruno to create scripts that generate and send test data to endpoints, which will be useful for testing the new touch data.
Screen Lock/Unlock Event: David Foster requested an event for when the user locks and unlocks the screen, which Warrd Adlani believes is possible by treating the SDK as separate from the keyboard, operating at the application level. The team may need to require users to add the SDK package to both the application and the keyboard extension to enable comprehensive functionality. This setup would allow a single session to be tracked across both the application and the keyboard extension.
